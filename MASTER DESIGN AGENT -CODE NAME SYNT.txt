20 gaps. 5 are critical — they break in production right now.

The most dangerous one is the matrix iterations. The workflow has strategy: matrix: iteration: [1,2,3,4,5] — GitHub Actions will fire all 5 jobs simultaneously. They all check out the same branch, make different changes, and try to push at the same time. You'll get merge conflicts and corrupted commits on the first real run. It needs to be sequential with needs: dependencies chaining each iteration to the previous one.

The second critical one is rollback. If iteration 3 predicts a score of 9.2 but iteration 4's Claude call goes wrong and outputs broken CSS, the workflow still merges it. There's no "compare to last known good" gate.

The brand token preservation gap is the sneakiest — because the upgrade prompt says things like "replace Inter with Bricolage Grotesque" without first reading whether the project has KUPURI's --bg: #edf1e8 or --accent1: #ed6a5a defined. It will overwrite them. The fix is a pre-upgrade step that reads globals.css, extracts all --custom-property values, and injects them as a "DO NOT CHANGE THESE" block into the upgrade prompt.

The copy gap is real too — right now the workflow flags happy talk but the upgrade job doesn't actually rewrite it with P.A.S.S. structure. It notes the issue in the commit message but leaves the original text. That's why a site could score 9.5 on design axes but still have "Welcome to our amazing platform" sitting in the hero.

Let me break this down in plain terms, no jargon.
What a CVE Is

CVE stands for Common Vulnerabilities and Exposures. It's a public registry — like a police report database, but for software security holes. When a researcher finds a way to break into a piece of software, they report it, it gets a CVE number, and it becomes public record. The number looks like CVE-2026-25253. That number means: discovered in 2026, the 25,253rd reported vulnerability that year.

Having a CVE doesn't mean you're hacked. It means someone found the door unlocked and told the world about it.
What an Exposed Instance Means

An "instance" is just a running copy of the software. If you installed OpenClaw on your laptop and it's running, that's one instance. If a company installed it on 50 servers, that's 50 instances.

An "exposed instance" means that copy is reachable from the open internet — not protected behind a firewall, not locked to a private network, not requiring a VPN to reach. It's sitting on a public IP address where anyone in the world can try to talk to it.

OpenClaw was designed to be convenient to set up. That convenience meant a lot of people installed it and left it accessible from the internet without realizing the risk. Researchers found 135,000 instances just sitting open like that.
What CVE-2026-25253 Actually Did

This is the bad one. One-click remote code execution.

Here's what that means in a real scenario. Say your OpenClaw instance is running on a server and accessible from the internet. Someone discovers this CVE. They don't need your password. They don't need to trick you into clicking anything. They send one crafted HTTP request to your server — a single packet of data — and OpenClaw executes whatever code they put inside it. On your machine. As whatever user OpenClaw is running as.

If OpenClaw is running as root (which many default installs do), they now have full control of that machine. Every file. Every credential stored in memory. Every API key OpenClaw has access to. Every tool it has been given permission to use.

The reason this is especially dangerous with an agent like OpenClaw is the blast radius. A normal web server getting hacked is bad. An agent server getting hacked is catastrophic. OpenClaw has been given permission to read files, write files, make API calls, send messages, interact with other services. The attacker doesn't just get the server — they inherit every permission the agent was given.
The Other Six CVEs in Plain Terms

Command injection — you send a specially crafted input and instead of OpenClaw treating it as data, it treats part of it as a system command and runs it. Like typing ; rm -rf / into a search box that was never supposed to run commands.

SSRF (Server-Side Request Forgery) — you trick OpenClaw into making HTTP requests on your behalf, to internal addresses it should never be able to reach. Cloud servers often have a special internal address (169.254.169.254) that gives out credentials for cloud services. SSRF lets an attacker ask OpenClaw to fetch that address and hand back the credentials.

Authentication bypass — there was supposed to be a login or API key check. This vulnerability skips it entirely. The door has a lock but someone found you can just push it open anyway.

Path traversal — OpenClaw is only supposed to read files in certain directories. This vulnerability lets you give it a path like ../../etc/passwd — the .. moves up out of the allowed directory into places it should never touch.
